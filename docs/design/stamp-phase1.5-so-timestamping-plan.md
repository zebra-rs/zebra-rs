# STAMP Phase 1.5 — Sender `SO_TIMESTAMPING` for accuracy

> **Status:** implementation plan, awaiting review (2026-06-13)
> **Parent docs:** [stamp-phase1-implementation-plan.md](./stamp-phase1-implementation-plan.md)
> (the shipped measurement plane), [bfd-sbfd-stamp-xdp-offload-notes.md §9b.3](./bfd-sbfd-stamp-xdp-offload-notes.md)
> (the accuracy ladder — this is **rung 1**, optionally **rung 2**).
> **Branch:** `stamp-so-timestamping`

---

## 1. Why

Phase 1 stamps T1/T4 in userspace. The D1 error budget
([offload notes §9b.1](./bfd-sbfd-stamp-xdp-offload-notes.md)) is:

```
delay_est = (fwd_wire + rev_wire)/2  +  (a+b+c+d)/2
```

where `d` = wire → sender T4-stamp and `a` = sender T1-stamp → wire are the
two **sender-side** residues. In a tokio daemon each RX boundary (`d`) is a
softirq→socket-queue→epoll-wake→task-poll chain — tens of µs idle, **ms-class
tails under load**. Those tails land in **max-delay and delay-variation**
(min-delay self-filters via the window minimum). Kernel `SO_TIMESTAMPING`
moves the T4 (and T1′) stamp into the kernel, removing the daemon-scheduling
part of `d` (and most of `a`) with no eBPF.

This is the parent plan's recommended first post-Phase-1 follow-up
([offload notes §9b.5 step 2](./bfd-sbfd-stamp-xdp-offload-notes.md)). The
single-T1-build-site / single-T4-read-boundary seams (Phase-1 R3) were built
for exactly this swap-in.

## 2. Scope

**Rung 1 (core, this PR):** software RX `SO_TIMESTAMPING` → kernel-stamped
**T4** on each session's connected sender socket, and kernel-stamped **T2** on
the reflector socket. Software timestamps are generated by the kernel network
stack at skb receive on **every** interface including veth/loopback (unlike HW
timestamps, rung 4), so this works in the BDD. Graceful fallback to `now_ntp()`
when no cmsg arrives (older kernel, option rejected).

**Rung 2 (optional, gated on review — may split to a follow-up PR):** software
TX `SO_TIMESTAMPING` + `MSG_ERRQUEUE` (`SOF_TIMESTAMPING_OPT_ID` keyed by seq)
→ corrected **T1′** retrieved after send, retained per-session and used in
*our* math. The in-packet T1 still goes out unchanged for the reflector's
echoed copy; only our `(T4−T1)` term swaps T1→T1′. Removes most of `a`.

**Out of scope:** rung 3 (XDP sender-RX fastpath), rung 4 (HW timestamps) —
both need the shared-supervisor refactor / real NICs (offload notes §9b.4,
§9b.5 stages B/C). No wire/protocol change in either rung — pure local
accuracy.

## 3. Design decisions

| # | Decision | Rationale |
|---|----------|-----------|
| T1 | Enable `SOF_TIMESTAMPING_RX_SOFTWARE \| SOF_TIMESTAMPING_SOFTWARE` (rung 1) via `setsockopt(SO_TIMESTAMPING)` on both the sender and reflector sockets | Software RX path; no NIC dependency, works on veth. `SOFTWARE` = report software stamps. |
| T2 | Parse `SCM_TIMESTAMPING` cmsg (`ControlMessageOwned::ScmTimestampsns` → `Timestamps.system` timespec); convert to NTP; **fall back to `now_ntp()` when absent** | Honest degradation — a kernel that doesn't deliver the cmsg must still measure (at rung-0 accuracy), not break. |
| T3 | Factor the UNIX→NTP epoch shift in `now_ntp()` into `unix_to_ntp(secs, nanos)`; reuse for the kernel-timestamp conversion | One conversion, unit-tested once. |
| T4 | `sender_read` switches from plain `recv` to `recvmsg` with a `SCM_TIMESTAMPING` cmsg buffer; carries a `t4_kernel: bool` up in `Message::ReplyRecv` for the counter | Same recvmsg shape the reflector already uses; the flag makes the feature observable. |
| T5 | Per-session counters `t4_kernel` / `t4_userspace`; surface in `show stamp statistics` (and the reflector's `t2_kernel`/`t2_userspace`) | Observable + BDD-assertable that kernel stamps are live on veth; no accuracy claim is BDD-checkable, but *delivery* is. |
| T6 | (Rung 2) Per-session `pending_tx: BTreeMap<seq, StampTimestamp>` of corrected T1′, capped/aged so unanswered probes don't leak; on `ReplyRecv` look up T1′ by `reply.sender_seq`, fall back to `reply.sender_timestamp` on miss | Late pairing — errqueue T1′ may arrive after or before the reply; bounded memory; miss-tolerant. |
| T7 | (Rung 2) The in-packet T1 stays userspace `now_ntp()` at build; only the *locally-retained* T1′ feeds our `(T4−T1)` | The reflector echoes the in-packet value; the wire is unchanged so interop is untouched. |

## 4. Step-by-step

### Step 0 — branch (done: `stamp-so-timestamping`)

### Step 1 — timestamp conversion (`stamp/timestamp.rs`)
- Extract `fn unix_to_ntp(secs: u64, nanos: u32) -> StampTimestamp` from
  `now_ntp`; `now_ntp` calls it. Add `fn timespec_to_ntp(ts: &TimeSpec)` (or
  take `(i64, i64)` to stay nix-agnostic).
- Unit tests: a known timespec → expected NTP words; `now_ntp` unchanged.

### Step 2 — socket options (`stamp/socket.rs`)
- New `set_so_timestamping_rx(&Socket)` helper: `setsockopt(SOL_SOCKET,
  SO_TIMESTAMPING_NEW or SO_TIMESTAMPING, RX_SOFTWARE|SOFTWARE)`. Prefer the
  nix typed `sockopt::Timestamping` if it exposes the flags cleanly; else raw
  `libc::setsockopt` mirroring the existing `set_ipv4_recvttl` pattern.
- Call it in **both** `stamp_reflector_socket` and `stamp_sender_socket`.
  Non-fatal on `EINVAL` (warn once) so an old kernel still runs at rung 0.

### Step 3 — receive paths (`stamp/network.rs`)
- `sender_read`: `recv` → `recvmsg::<()>` with `cmsg_space!(ScmTimestampsns)`;
  walk cmsgs for `ScmTimestampsns`; `t4 = system→ntp` else `now_ntp()`; carry
  `t4_kernel: bool` in `Message::ReplyRecv`.
- `reflector_read`: it already `recvmsg`s — widen the cmsg buffer to include
  `ScmTimestampsns`, prefer the kernel `system` stamp for `rx_ts` (T2), else
  the current `now_ntp()`; carry `t2_kernel: bool` in `Message::ProbeRecv`.

### Step 4 — instance plumbing (`stamp/inst.rs`)
- `Message::ReplyRecv { …, t4_kernel }` and `Message::ProbeRecv { …, t2_kernel }`.
- `Session`: `t4_kernel: u64`, `t4_userspace: u64`. `ReflectorStats`:
  `t2_kernel`, `t2_userspace`. Bump in `on_reply_recv` / `on_probe_recv`.
- **(Rung 2 only)** `pending_tx` map on `Session`, fed by the errqueue reader
  (Step 6); `on_reply_recv` resolves T1′.

### Step 5 — show (`stamp/show.rs`)
- `show stamp statistics`: add "Kernel RX timestamps: T4 N/M, T2 N/M" lines
  (kernel/total), JSON fields alongside.
- `show stamp session` detail: per-session `t4_kernel`/`t4_userspace`.

### Step 6 — (Rung 2, optional) TX errqueue
- `stamp_sender_socket`: add `SOF_TIMESTAMPING_TX_SOFTWARE |
  SOF_TIMESTAMPING_OPT_ID | SOF_TIMESTAMPING_OPT_TSONLY`; track an OPT_ID
  counter aligned with `next_seq`.
- New `sender_errqueue_read(key, sock, tx)` task draining `MSG_ERRQUEUE`;
  pairs the returned `ee_data` (OPT_ID = seq) with the SCM_TIMESTAMPING stamp;
  sends `Message::TxStamp { key, seq, t1_prime }`. Instance stores into
  `pending_tx` (aged).

### Step 7 — tests
- Unit: `unix_to_ntp` / `timespec_to_ntp` round-trip; counter increments;
  (rung 2) pending-map pairing + miss fallback + aging.
- Integration: extend the loopback test to assert the reply's T4 came from the
  kernel path (loopback delivers software RX stamps) — i.e. `t4_kernel > 0`
  after the first export.
- BDD: extend `@stamp_te_metric` with one assertion that `show stamp
  statistics` reports kernel T4 > 0 (veth delivers software RX stamps). No
  accuracy assertion — precision isn't BDD-checkable; delivery is.

### Step 8 — verify & deliver
- `cargo test --workspace --exclude bdd`, `cargo fmt`, `cargo clippy
  --workspace --all-targets -- -D warnings`.
- Install binary + YANG (md5 before/after), `make stamp_te_metric` + one
  regression feature; PR.

## 5. Risks

| Risk | Mitigation |
|------|------------|
| nix 0.31 API surface for `SO_TIMESTAMPING` differs from assumption | Step 2/3 verify the exact `sockopt::Timestamping` / `ScmTimestampsns` names at implementation; raw-libc fallback (we already use it for `IP_RECVTTL`). |
| `SO_TIMESTAMPING_NEW` vs legacy on 32-bit time_t | Target is x86-64/aarch64 (64-bit time_t); use the value the running kernel accepts, fall back on EINVAL. |
| Software RX stamp not delivered in some namespace/kernel | Fallback to `now_ntp()` per T2; counter shows 0 kernel, feature is inert not broken. |
| Rung 2 errqueue complexity (late pairing, leaks) | Bounded aged `pending_tx`; ship rung 1 first, gate rung 2 on review — may be a separate PR. |

## 6. Recommendation

Ship **rung 1** as the core PR (the recommended first follow-up; ~tens of
lines + counters + tests). Decide rung 2 inclusion at review time — it is
moderate plumbing and cleanly separable into a fast-follow if rung 1 should
land thin.
